# SpecDev Workflow Issues — Brainstorm-Phase Friction

**Date:** 2026-05-10
**Context:** A coding agent (Claude) drove a brainstorm phase for assignment `00031_feature_research-workflow-guardrails` in the oceanshed-cli repo. The phase produced valid `proposal.md` and `design.md`, but the session surfaced **sixteen** issues with the SpecDev workflow guide and core skills, captured below in three groups: four from direct in-session friction, six from a structured cross-file audit (Appendix A), and six from a "weak-agent literal-reading" audit (Appendix B). Each is verified against the actual files and includes a suggested fix.

The issues are ranked by impact, not by severity. They're observations from one session; further sessions may surface or invalidate them.

## 1. No "verify against current code" gate before drafting design sections

**What happened:** The brainstorming skill (`.specdev/skills/core/brainstorming/SKILL.md`) Phase 1 step 3 says to "use `rg` for current implementation facts." Phase 3 (Design Sections) does not require verification before drafting. The agent drafted a Goals section that cited `signal_validator.py:320`, `sweep_runner.py:194`, and several other file:line locations — all paraphrased from the source issues document and prior knowledge, none verified against the actual current code.

The user had to stop the flow with **"before we continue, check everything against the existing code"**. That single correction caught two real misalignments:

- The agent's draft proposed creating a new `selected.toml` artifact for sweep-layer parameter reuse. Verification against `scripts/workspace_tool.py:639` revealed that `decision.toml` already exists with the right shape — only the *consumer* side was missing. The design changed from "create new artifact" to "consume existing artifact."
- The agent's draft proposed adding `python` and `created_at` fields to `signal_runner.run_metadata.json`. Verification against `scripts/signal_runner.py:188` revealed those fields already existed. The design changed to "extend `sweep_runner.run_metadata.json` to match the existing `signal_runner.run_metadata.json` shape."

Both are non-trivial scope corrections that would have led to wasted implementation work if the user hadn't intervened.

**How the workflow failed:** The skill treats verification as an implicit responsibility of Phase 1 ("Understand"), but Phase 3 ("Design Sections") naturally invites the agent to draft based on what it remembers from Phase 1. There is no checkpoint between "I understand the problem" and "here is a design citing concrete file:line locations" that requires re-verifying citations. The `verification-before-completion.md` always-apply skill is about *post-implementation* claims (tests pass, build succeeds), not about *pre-design* citations.

**What should happen:** Add an explicit Phase 3 step 0 to the brainstorming skill:

> Before presenting any section that cites a file, line, symbol, or behavior, verify each citation against the current code with `rg` or `cat`. Either confirm the citation or revise it before presenting. Do not present a section whose citations have not been verified in this session.

This makes verification mechanical rather than relying on the agent's discipline. It also gives the user a clear handle for stopping unverified content: "did you verify this section?"

## 2. The brainstorming skill contradicts itself on MCQ vs bullet lists

**What happened:** Phase 1 step 4 of the brainstorming skill says:

> Ask the user 1-3 tightly related questions per message to understand their goal
>
> - **Prefer multiple-choice over open-ended**

Phase 4 step 5 then literally models a bullet list at the close of the brainstorm:

```
5. Tell the user their options:
   - `specdev reviewloop brainstorm` — automated external review
   - `specdev review brainstorm` — manual review in a separate session
   - `specdev approve brainstorm` — skip review and proceed to breakdown
```

The agent followed the Phase 4 model rather than the Phase 1 rule, and presented the brainstorm-close as a plain bullet list. The user immediately flagged: **"how come you didnt follow the specdev to present above as multiple choice?"** They were right — the discrete close decision (review-loop vs manual-review vs approve) is exactly the kind of decision MCQ is for.

**How the workflow failed:** The skill text contradicts itself. The agent saw the Phase 4 bullet list, treated it as authoritative ("the skill is showing me how to present this"), and missed that it conflicts with the Phase 1 rule. A skill that contradicts itself trains the wrong behavior, even when the rule is stated correctly elsewhere.

**What should happen:** Pick one of two fixes:

**Option A (preferred):** Make Phase 4 step 5 model the rule. Replace the bullet list with: "Present the close as a multiple-choice question via the agent's question-asking interface, with the three options as choices. Do not present them as a bullet list."

**Option B:** Carve the rule back to Phase 1 explicitly: "Prefer multiple-choice for understanding-phase questions. Phase 4 close options may be a bullet list because no answer is required." This is weaker — discrete decisions deserve MCQ regardless of phase — but at least removes the contradiction.

Either fix is better than the current state where the skill says one thing and models another.

## 3. `_workflow_feedback/` is hidden during brainstorm

**What happened:** The repository's `.specdev/knowledge/_workflow_feedback/` folder contains real gotchas from prior sessions:

- `breakdown-checkpoint.md` — "This CLI does not support `specdev checkpoint breakdown`"
- `reviewloop-flag-parsing.md` — "Use equals-form reviewer flags: `--reviewer=claude` not `--reviewer claude`"
- `reviewloop-stale-current.md` — "After approved review loop, `.specdev/.current` may point at a stale assignment name"
- `reviewloop-artifact-fallback.md` — "reviewloop can produce a valid review in stdout but fail to persist the expected feedback round"
- `subagent-git-conflicts.md` — "do NOT let parallel subagents run `git commit`"

The brainstorming skill Phase 1 step 3 says to search via `specdev knowledge search "<topic keywords>"`. Those underscore-prefixed feedback files never surfaced for the agent's queries (`"sweep parameter selection provenance"`, `"data availability ticker oceandata"`, `"valid_end_date signal_info"`, etc.) because they're not topic-tagged — they're process gotchas about the SpecDev CLI itself.

The agent did not read any of these files during the brainstorm. None bit during this phase, but they would have during breakdown or implementation, costing real time.

**How the workflow failed:** Topic-search-based knowledge discovery is the wrong model for process gotchas. A gotcha about `specdev reviewloop` flag parsing is relevant to *every* assignment that uses reviewloop, regardless of the assignment's topic. It's not discoverable via topic search and shouldn't have to be.

**What should happen:** Add a step at the start of every phase skill — not just brainstorm — that says:

> Before starting this phase, read every file in `.specdev/knowledge/_workflow_feedback/`. These are SpecDev workflow gotchas that apply to every assignment. They are not topic-tagged and will not surface in `specdev knowledge search`.

Alternatively, integrate this as a CLI step: `specdev assignment` could print the contents of `_workflow_feedback/` summaries when reserving an ID. Either way, hiding known gotchas behind a search query the agent doesn't know to make is a workflow bug.

## 4. Section-size guidance doesn't match real complexity

**What happened:** The brainstorming skill Phase 3 says:

> 2. Present one section at a time (200-300 words each)

For assignment 00031, the natural section sizes were:

- Goals: ~450 words (six numbered goals across four code surfaces)
- Non-Goals: ~400 words (eight explicit boundaries)
- Design: would have been ~2000 words as a single section; the agent organically split it into Streams 1–4 of ~300 words each

The agent split the Design section without explicit permission from the skill. This worked, but the skill's "200-300 words each" rule provides no escape hatch. A literal reading would force either oversized sections or arbitrary truncation.

**How the workflow failed:** The word count is presented as fixed when it should be a guideline. Real assignments span a wide complexity range. A bugfix may have a 50-word Root Cause; a multi-stream feature may have a 2000-word Design. The skill should describe the goal (validate incrementally) rather than dictate a fixed word budget.

**What should happen:** Replace the word-count rule with a structural rule:

> Present sections one at a time. After each section, ask: "Does this look right so far?" If a section would exceed roughly 300 words, split it into named sub-sections and present each separately. Section length should match the section's natural complexity, not a fixed budget.

This matches what the agent did organically and removes the rule's brittleness.

## Summary

The four issues cluster around two root causes:

**Discipline gaps in the brainstorming skill itself.** Issues #1 (no verify-before-design gate) and #2 (MCQ rule vs bullet model) are both cases where the skill text doesn't enforce its own intent. The agent has to read between the lines or be corrected by the user.

**Discoverability of operational knowledge.** Issues #3 (workflow feedback hidden) and #4 (rigid word counts) are both about how the framework surfaces guidance. Process gotchas are buried behind topic search; structural rules are over-specified instead of describing intent.

Both root causes are tractable with small, targeted edits to `.specdev/skills/core/brainstorming/SKILL.md` and `.specdev/_guides/workflow.md`. None require redesigning the framework. The highest-impact single fix is #1 (verify-before-design gate) — it would have caught the two real scope errors in 00031 without any user intervention.

# Appendix: Cross-File Audit Findings

The four issues above came from one brainstorm session. After writing them up, the same agent then ran a structured cross-file audit of `/mnt/h/oceanwave/lib/cli/oceanshed-cli/.specdev/` (all `_guides/`, `skills/core/*/SKILL.md`, `_templates/`, `_main.md`, `_index.md`) looking for contradictions, ambiguities, dead references, and self-referential traps. Six additional issues were found and verified against the actual files. Each is independent of the four above.

The six are ordered by impact.

---

## 5. Two parallel `workflow_feedback` directories — undocumented one holds 5 critical gotchas

**What's the issue:** `.specdev/knowledge/` contains both `workflow_feedback/` and `_workflow_feedback/`. They hold different files:

- `knowledge/workflow_feedback/` — 1 file (`implementation-review-selection.md`).
- `knowledge/_workflow_feedback/` — 5 files (`breakdown-checkpoint.md`, `reviewloop-flag-parsing.md`, `reviewloop-stale-current.md`, `reviewloop-artifact-fallback.md`, `subagent-git-conflicts.md`).

Every documentation reference points to the no-underscore path:

- `_templates/gate_checklist.md:76` — `Workflow observations noted in knowledge/workflow_feedback/`
- `skills/core/knowledge-capture/SKILL.md:87` — `Write workflow observations to knowledge/workflow_feedback/`

The underscore-prefixed directory is referenced nowhere in `_main.md`, `_index.md`, `_guides/*`, `skills/`, or `_templates/`. Yet it contains the five most operationally-important workflow gotchas — the kind of process knowledge a new agent absolutely needs (e.g., "use `--reviewer=claude` not `--reviewer claude` or the CLI crashes").

**Why it matters:** Agents writing new feedback follow the documented path (`workflow_feedback/`) and never see the underscore-prefixed corpus. That directory is invisible to topic search and to any documentation-driven discovery. The five gotchas it contains are real bugs that have already burned past sessions, and new agents are doomed to re-hit them. This is the most severe finding in the whole audit because the gap is between "documented surface" and "actual operational knowledge."

**Suggested fix:** Pick one canonical path. Either move all five underscore-prefixed files into `workflow_feedback/` and delete the underscore directory, or rename the documented references to the underscore path. Then add a step to every phase skill: "Read every file in `knowledge/workflow_feedback/` at phase start." Connects directly to original issue #3.

## 6. Breakdown task regex `^### Task [0-9]` silently drops tasks 10+

**What's the issue:** `skills/core/breakdown/SKILL.md:66` says:

> Every task MUST be an H3 heading (`### Task N: …`). The breakdown scripts grep for `^### Task [0-9]` — H2 (`## Task N:`) silently produces zero tasks and breaks downstream tracking.

The regex `[0-9]` matches a single digit. Tasks 1–9 work; tasks 10+ silently fail to match. The text doesn't acknowledge this — it presents the regex as the documented contract.

**Why it matters:** Any plan with 10 or more tasks loses tasks 10+ from downstream tracking. The failure mode is silent: the breakdown checkpoint succeeds, the plan looks complete in markdown, but task 10 never reaches the implementing skill's task list. For complex assignments — exactly the ones most likely to need careful tracking — the tooling underreports work.

**Suggested fix:** Either (a) update the script to use `^### Task [0-9]+`, or (b) if the underlying script can't be changed, document the limit explicitly: "Tasks must be numbered 1–9. If you need more tasks, split the assignment." (a) is strongly preferred.

## 7. `LOW / MEDIUM / HIGH` complexity required by gate but undefined anywhere

**What's the issue:** `_templates/gate_checklist.md:11` requires:

> - [ ] Complexity class set: LOW / MEDIUM / HIGH

Nothing else in `.specdev/` defines what those levels mean — no thresholds, no examples, no decision tree. Searching `_guides/`, `skills/`, `_templates/`, `_main.md`, and `_index.md` for `LOW.*MEDIUM.*HIGH`, `complexity class`, or `Complexity:` returns only the gate-checklist line itself.

**Why it matters:** Every assignment must produce this classification, but every classification is a guess. Two agents will classify the same assignment differently. If anything downstream conditions on complexity (e.g., scaffolding requirements, architecture review gates), the conditioning is meaningless because the input is uncalibrated.

**Suggested fix:** Add a "Complexity Classification" section to `_guides/workflow.md` or `_guides/assignment_guide.md` with concrete criteria: e.g., "LOW: 1–3 tasks, single module; MEDIUM: 4–7 tasks, 2–3 modules, no new architecture; HIGH: 8+ tasks or new architecture." Or, if complexity isn't actually used by anything downstream, remove the gate item.

## 8. Phase 4 named three different ways — "Summary", "Capture", "knowledge-capture"

**What's the issue:** The same phase has three names used inconsistently:

- `_main.md:30` — "Summary (capture)"
- `_guides/workflow.md:67` — "Phase 4: Summary (Capture)"
- `skills/README.md:19` — "knowledge-capture"
- All skill `next:` headers (implementing/SKILL.md:8, investigation/SKILL.md:8, etc.) — "knowledge-capture"
- The skill folder itself — `skills/core/knowledge-capture/`
- Hook output (observed this session) — "Phase: summary"

**Why it matters:** Agents reading different documents form different mental models of the phase. Test/script code that conditions on phase name (e.g., the SessionStart hook printing `Phase: summary`) drifts from the names skills use internally. Over time, conditional logic and documentation references diverge.

**Suggested fix:** Pick one canonical name and apply it everywhere. "knowledge-capture" matches the folder and the skill name and is the most descriptive. Update `_main.md:30`, `_guides/workflow.md:67`, and any tooling that prints `summary`.

## 9. Plan review (Phase 5 of breakdown) has no termination rule

**What's the issue:** `skills/core/breakdown/SKILL.md` documents two internal reviews. Design review (Phase 1) has a termination rule:

- Line 36: `Re-review (max 2 rounds total)`
- Line 37: `If still failing after 2 rounds: surface findings to user and pause`

Plan review (Phase 5) does not:

- Line 136: `A subagent review (1-2 rounds) will check the plan for completeness and correctness`
- Line 137: `If the review finds issues: address them and re-run the review`

There is no upper bound on how many times plan review can re-run, no escalation path, and no instruction to surface to the user. Combined with Phase 5's later "Do NOT stop, report, or wait for user input" (lines 140–142), an agent can lock itself into an indefinite re-review loop on a fundamentally unfixable plan.

**Why it matters:** Same failure mode as design review without the safety valve. An assignment whose design has structural issues (wrong architecture, missing success criteria) might survive Phase 1 but fail Phase 5 review repeatedly. Without a max-rounds rule, the agent burns time and tokens until the user intervenes.

**Suggested fix:** Add to Phase 5: "Plan review max 2 rounds. If still failing after 2 rounds, surface findings to user, recommend revising design, and pause." Match the Phase 1 design-review safety rule.

## 10. "Phase 2.5" in breakdown breaks the integer phase-numbering convention

**What's the issue:** `skills/core/breakdown/SKILL.md` numbers its internal phases sequentially — Phase 1, 2, 3, 4, 5 — with one exception: line 54 introduces `Phase 2.5: Choose Execution Mode`. This is the only fractional phase number anywhere in `.specdev/`.

**Why it matters:** Lower impact than the others, but real. Tooling or skills that parse phase numbers as integers (e.g., progress trackers, conditional breakdowns) special-case this one entry. Documentation that says "after phase 2 do X" is ambiguous about whether 2.5 counts as still in phase 2 or already past it. The fractional name signals "this was added later and didn't fit the existing structure" — a smell of accreted complexity.

**Suggested fix:** Either rename to `Phase 2b` (still signals "between 2 and 3" but uses a non-numeric suffix that won't parse as 2.5), or fold the execution-mode choice into the end of Phase 2 as the final step.

## Appendix Summary

The six new issues fall into three buckets:

- **Hidden operational knowledge:** issue #5 (parallel `workflow_feedback` dirs) is a serious gap between documentation and actual gotcha corpus. Fixing this alone would prevent multiple recurring sessions of re-discovering the same bugs.
- **Concrete bugs in tooling contracts:** issues #6 (single-digit task regex) and #9 (plan re-review with no max rounds) are mechanical bugs with silent failure modes.
- **Definitional drift:** issues #7 (undefined complexity levels), #8 (phase-4 naming), and #10 (Phase 2.5) are terminology and contract drift that compound over time.

Combined with the original four issues, the audit found ten distinct workflow guide issues in one session. None require redesigning SpecDev; all are addressable with targeted edits to specific files. The single highest-leverage fix remains original issue #1 (verify-against-current-code gate) plus appendix issue #5 (canonical workflow_feedback path).

# Appendix B: Dumb-Agent Failure Modes

The previous ten issues assumed a capable coding agent that can infer intent from context. SpecDev is also used by weaker agents that take instructions literally, can't disambiguate overloaded terms, and won't notice when two adjacent paragraphs contradict each other. This appendix re-audits the same files asking: **where would a dumb agent break?**

Six failure modes were found and verified. They aren't bugs in the SpecDev framework — they're places where the workflow assumes the agent will fill in implicit knowledge. A capable agent does that fill-in silently and the workflow runs fine. A weaker agent stalls, guesses wrong, or produces malformed output.

## 11. Brainstorming SKILL.md and the design template specify different `design.md` sections

**What's the issue:** Two adjacent files name different required sections for `brainstorm/design.md`:

- `skills/core/brainstorming/SKILL.md:85-91` says the design must include: *Goal and approach, Architecture and components, Key decisions with reasoning, Success criteria, Testing approach.*
- `_templates/brainstorm-design.md:6-11` says (for `feature`): *Overview, Goals, Non-Goals, Design, Success Criteria.*

Only "Success Criteria" overlaps between the two lists. Five other section names differ.

**Why it matters for a dumb agent:** A capable agent reads both, notices the divergence, and produces something containing all required content under names that satisfy both lists (which is what this session's design.md does — it has both `Goals` and `Success Criteria` to bridge). A dumb agent picks one source as authoritative and produces a design that fails the other source's checklist. Phase 1 of breakdown then either accepts a non-conforming design or — if it follows the SKILL.md side — escalates to the user about a "missing Testing Approach section" that the template never required.

**Suggested fix:** Pick one canonical section list and apply it to both files. Probably the template's list (`Overview, Goals, Non-Goals, Design, Success Criteria`) since the template is what the agent actually copies into `design.md`.

## 12. The word "Mode" is overloaded across two independent taxonomies

**What's the issue:** Two unrelated concepts share the name "mode":

- **Plan execution mode** (`breakdown/SKILL.md:54-62`, "Phase 2.5: Choose Execution Mode") — values are `inline`, `subagent`, `parallel`. Set once in the plan header. Tells `implementing` *how* to dispatch tasks.
- **Per-task mode** (`breakdown/SKILL.md:92-104`, in Phase 3) — values are `standard`, `full`, `lightweight`. Set per task. Tells `implementing` whether a reviewer subagent is dispatched after the implementer.

The implementing skill compounds the ambiguity:

- Phase 1 step 4 (`implementing/SKILL.md:43-47`) says "Resolve execution mode from the plan header" — that's the plan-level mode.
- Phase 2 step 3 (`implementing/SKILL.md:62-65`) says "Mode-based review — use the `mode` from step 1" — but "step 1" is `prepare-task.sh`, which returns the *task-level* mode.

So within ten lines of each other, `implementing` uses "mode" to mean two different things.

**Why it matters for a dumb agent:** A capable agent tracks which "mode" a sentence is referencing from context. A dumb agent confuses them. Likely failure: it sees the plan header `Execution Mode: inline`, starts the first task, runs `prepare-task.sh`, gets back `mode: full`, and either ignores the new mode (silently skipping reviewer dispatch on full-mode tasks) or treats `full` as if it overrode the plan-level execution mode.

**Suggested fix:** Rename one of them. "Execution mode" can stay; the per-task mode could become "review mode" or "task class". Both files should consistently use whichever name they pick, and the JSON returned by `prepare-task.sh` should match.

## 13. Plan-header template uses placeholders but the example plan.md uses a different shape entirely

**What's the issue:** `breakdown/SKILL.md:117-132` shows the plan header template:

```
# [Feature Name] Implementation Plan
> **For agent:** Implement this plan task-by-task using TDD discipline.
**Goal:** [One sentence from design]
**Architecture:** [2-3 sentences from design]
**Tech Stack:** [From design]
**Execution Mode:** inline
```

The example at `_templates/assignment_examples/feature/00000_feature_email-validator/breakdown/plan.md` opens like this:

```
# Implementation Plan: Email Validator Utility
## Summary
Create a simple email validation utility module ...
## Complexity Gate
- Class: LOW
- Rationale: ...
## Tech Stack
...
## Architecture
### Module Structure
```

Different title format. No `For agent:` line. No bold inline `Goal:`/`Architecture:`/`Tech Stack:` block. No `Execution Mode:` field. The example uses `## Summary`, `## Complexity Gate`, `## Tech Stack`, `## Architecture` as H2 sections instead.

**Why it matters for a dumb agent:** A capable agent sees both, picks one, and the breakdown CLI scripts probably tolerate either because they only grep for `^### Task [0-9]`. A dumb agent picks the example (because it's a complete file rather than a fragment) and misses the `Execution Mode:` field — which means `implementing/SKILL.md` Phase 1 step 4 can't resolve the mode and falls into the "Missing or unknown mode: treat as `inline`" branch. If the brainstorm intended subagent or parallel execution, that intent is silently lost.

**Suggested fix:** Make the example match the template, or update the template to look like the example, then verify both. Add a concrete *filled-in* example next to the template so dumb agents don't leave `[One sentence from design]` literal.

## 14. "as appropriate" / "if applicable" rules are unenforceable for a dumb agent

**What's the issue:** Eight occurrences of vague qualifiers across the workflow guidance, including critical ones:

- `knowledge-capture/SKILL.md:82` — "Write synthesized observations to `knowledge/` branches as appropriate"
- `knowledge-capture/SKILL.md:87` — "Write workflow observations to `knowledge/workflow_feedback/` as appropriate"
- `_templates/gate_checklist.md:68` — "`feature_descriptions.md` updated (if applicable)"
- `_templates/gate_checklist.md:76` — "Workflow observations noted in `knowledge/workflow_feedback/` (if applicable)"

The closing line `knowledge-capture/SKILL.md:88` makes the gap explicit: "If no new observations to write, that's OK — not every assignment produces reusable knowledge." So the rule is "write something, but only when appropriate, and 'nothing' is also a valid answer." There is no test a dumb agent can apply.

**Why it matters for a dumb agent:** A capable agent uses judgment: "this assignment had a real architectural decision; that's worth a knowledge file." A dumb agent has two equally-defensible defaults — write nothing every time, or write something every time — and the workflow accepts both. Result: either knowledge stops accumulating (dumb-skip), or `knowledge/` fills with low-signal entries that future searches have to filter through (dumb-write).

**Suggested fix:** Replace each "as appropriate" with a positive criterion the agent can mechanically apply. For example: "Write a knowledge file if the assignment introduced a new pattern, contradicted prior assumptions, or surfaced a workflow gotcha. Otherwise skip." Even a rough rubric is better than "appropriate."

## 15. Plan summaries in `complete-task.sh` have no length or format contract

**What's the issue:** `implementing/SKILL.md:66` says:

> Run `.specdev/skills/core/implementing/scripts/complete-task.sh <plan-file> <N> "<summary of task changes>"` … This marks the task as completed, stores the summary, and reports batch progress.

The "summary of task changes" string has no documented length, format, content requirements, or audience. Is it for the next agent in the batch? For the human reviewer? For knowledge-capture? Should it be one sentence or a paragraph? Should it list files changed, or describe behavior, or both?

**Why it matters for a dumb agent:** A capable agent writes a useful 1-2 sentence summary suited to whoever might read it. A dumb agent writes either "Done." (useless) or a 500-character recap that floods the batch report. There is no rule that catches either failure mode.

**Suggested fix:** Document the contract in `implementing/SKILL.md`: "Summary should be 1-2 sentences naming the behavior added or fixed and the user-visible files. Audience: human reviewer skimming a batch report. Example: `Added preflight warning for valid_end_date desync; updated signal_validator.py and added a fixture test in tests/preflight_warnings.test.js.`"

## 16. Phase 1 design review is simultaneously "lightweight sanity check" and "max 2 rounds with user escalation"

**What's the issue:** `breakdown/SKILL.md` describes the same review with two contradictory framings:

- Line 39: "This is a lightweight sanity check — the design was already validated section-by-section with the user during brainstorm."
- Lines 36–37: "Re-review (max 2 rounds total). If still failing after 2 rounds: surface findings to user and pause."

The second framing implies the review is heavyweight enough that 2 rounds may not suffice and user escalation is a real possibility.

**Why it matters for a dumb agent:** A capable agent reads "lightweight" as tone and "max 2 rounds, escalate" as the actual contract. A dumb agent could go either way:

- "Lightweight" → spend minimal effort on the review, miss real gaps, pass a thin design through to decomposition.
- "Max 2 rounds, escalate" → treat every minor issue as round-blocking, burn both rounds on cosmetic fixes, escalate to the user over things the brainstorm already settled.

The right behavior is somewhere between, but the skill provides no anchor.

**Suggested fix:** Pick a side. Either it's lightweight and we drop the escalation language (just fix in one pass and move on), or it's a real review and we drop "lightweight." A capable agent can navigate the contradiction; a dumb one lurches between extremes.

---

## Appendix B Summary

The six dumb-agent failure modes cluster around three root causes:

- **Same name, different things** (issues #11, #12) — when two concepts share a label, a literal-minded agent picks one definition and ignores the other.
- **Examples don't match templates** (issue #13) — when SKILL.md template and the example assignment diverge, a literal agent picks one and the other's contract silently fails.
- **Vague qualifiers as the only contract** (issues #14, #15, #16) — "appropriate", "if applicable", "lightweight" can't be checked; a literal agent either over- or under-applies the rule.

None of these issues block a strong agent. All of them block or degrade a weak one. For SpecDev to be portable across agent capability levels, the workflow text needs to be pickable-up by an agent that can't read between the lines.

If only one fix from this appendix lands, it should be #12 (rename one of the two "modes"). It's the simplest change, affects the highest-traffic files, and removes a confusion that compounds across every assignment.
